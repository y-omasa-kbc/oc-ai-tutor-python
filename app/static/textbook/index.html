1. はじめに

「AI に自分の資料を読ませて、その内容をもとに賢く答えさせる」── これから 45 分で、あなたはそんな AI を 自分の手で組み上げていきます。

使う技術は RAG(Retrieval-Augmented Generation)。 社内文書の検索、カスタマーサポート、専門家の相談窓口など、 いま世界中で導入が広がっている最前線の仕組み です。

2. RAG とは?

RAG は Retrieval-Augmented Generation の略で、日本語にすると「検索して、答えを作らせる」手法です。

次の 4 つのステップで動きます:

  1. ユーザーが 質問 を入力する
  2. AI が 資料 の中から、質問に合いそうなものを 検索 する
  3. 見つけた資料を LLM に渡す
  4. LLM が資料を読みながら 回答 を作る

普通の ChatGPT と違うのは、資料を渡してから答えさせる ところです。 今日はこの「資料を渡すところ」を、実際に 江戸時代の歴史 でやってみます。

3. RAG を試す

いま右側に出ている Web UI(画面に表示されているアプリ)が、今日あなたが動かす RAG です。 質問を入れると、AI が資料(「秀吉の天下統一」「関ヶ原への道」「江戸幕府の確立」)から答えを探して答えてくれます。

やってみよう:

  1. 隣のウィンドウでアプリを表示
  2. 下の例を入力して送信
  3. 「参考にした資料」の欄に どの資料が引かれたか が出るので確認

資料に答えがある質問(ちゃんと答えが返るはず):

  • 本能寺の変を起こした人物は?
  • 太閤検地とは何ですか?

資料に無い質問(「参考資料からは分かりません」と正直に返るはず):

  • 坂本龍馬って何をした人?
  • 明治維新のリーダーは誰?

資料に載っていないことは 正直に「分からない」 と答えるのが RAG の大事な性質です。 知ったかぶりをせず、資料をもとに答える。だから 資料の質 が答えの質を決めます。

4. 検索の落とし穴

この RAG は、質問の言葉 と 資料の中の言葉 がズレていると、 関係ない資料を引いてしまい、AI が「参考資料からは分かりません」と答えることがあります。 特に、私たちが普段使う カジュアルな言葉(「トップ」「仕切る」「リーダー」など)は 歴史の資料に出てこないので、うまくヒットしません。

試してみよう:

  • 「戦国時代のあと、日本を仕切ったのは誰?」

この質問は「参考資料からは分かりません」になるはずです。 資料には答えが書いてあるのに、AI が見つけられていません。

5. 質問改善(Query Transformation)を実装する

Query Transformation とは、AI に「この質問、資料ではこう書かれていそうだな」と考えさせて、 検索用のキーワード列 に書き直してから資料を探す仕組みです。

具体的には、LLM に「質問文から重要な言葉を抜き出して、関連する固有名詞を少し補って」と お願いする処理を入れます。

コピーするコード

質問の書き換えを担当するのが query_transformer.py というファイルです。 AI に対して「こう書き直してほしい」という 指示(プロンプト) を渡し、 書き直された質問を受け取る、という部品が入っています。

このファイルを開くと、def transform_query という関数(プログラムの部品)があります。 今の中身は 「質問をそのまま返す」だけになっています。これを下のコードで丸ごと置き換えます。 

def transform_query(query: str) -> str:
    if not query.strip():
        return query

    if os.getenv("DISABLE_QUERY_TRANSFORM", "0") == "1":
        return query

    messages = [
        {"role": "system", "content": QUERY_REWRITE_SYSTEM_PROMPT},
        {"role": "user", "content": query},
    ]
    try:
        rewritten = chat(messages=messages, temperature=0.0)
        return rewritten.strip() or query
    except Exception:
        return query

編集前と編集後の違い

編集前: 何もしていません。受け取った質問を、そのまま次の検索に渡すだけです。 

def transform_query(query: str) -> str:
    return query

編集後: 質問を、いったん AI に「検索しやすい言葉に書き直して」とお願いし、 戻ってきた結果を検索に渡すようになります。たったこれだけで、検索の当たり方が大きく変わります。

確認してみよう
さっき失敗した質問(「戦国時代のあと、日本を仕切ったのは誰?」など)をもう一度入れてみてください。 引かれる資料や回答が 変わる はずです。

6. システムプロンプトを変える

ここまでは「資料を探す」側を変えてきました。 ここでは「回答を作る」側の AI への指示、システムプロンプト をいじってみます。

同じ資料と質問でも、指示の書き方 で回答の雰囲気や内容が変わります。

回答を作るときの AI への指示は、prompts.py というファイルの SYSTEM_PROMPT にまとまっています。このファイルを開き、 SYSTEM_PROMPT = (...) の中身を、下のサンプルのどれかで丸ごと置き換えます。

雛形サンプル A: もっとカジュアルに

堅い歴史の文章を、友達に説明するような口調にしたいとき。

SYSTEM_PROMPT = (
    "あなたは江戸時代の歴史に詳しい、友達みたいな『歴史の家庭教師』です。"
    "ユーザーからの質問に対して、与えられた参考資料だけを根拠に、"
    "★中学生にもわかるよう、タメ口で、親しみやすく答えてください。"
    "回答は3〜6文程度の短さを心がけ、結論→理由→具体例 の順で話してください。"
    "参考資料に書かれていないことは推測せず、"
    "分からない場合は『ごめん、参考資料からは分からんわ』と正直に答えてください。"
)

雛形サンプル B: 初心者向けに噛み砕く

歴史をほぼ知らない相手に教える場面。

SYSTEM_PROMPT = (
    "あなたは江戸時代の歴史に詳しい、はじめての人に寄り添う『歴史の家庭教師』です。"
    "ユーザーからの質問に対して、与えられた参考資料だけを根拠に、"
    "★中学1年生にもわかる言葉で、むずかしい漢字は開き直さず、易しい言葉に置き換えて答えてください。"
    "回答は3〜6文程度の短さを心がけ、結論→理由→具体例 の順で話してください。"
    "参考資料に書かれていないことは推測せず、"
    "分からない場合は『参考資料からは分かりません』と正直に答えてください。"
)

雛形サンプル C: ドラゴンボールの孫悟空になりきる

回答の口調を大胆に変えて、キャラクターになりきらせたいとき。

SYSTEM_PROMPT = (
    "あなたは江戸時代の歴史に詳しい『歴史の家庭教師』ですが、話し方は『ドラゴンボール』の孫悟空になりきってください。"
    "ユーザーからの質問に対して、与えられた参考資料だけを根拠に、"
    "★一人称は「オラ」、二人称は「おめえ」、語尾は「〜だぞ」「〜だな」で、素朴で元気な田舎の言葉づかいで答えてください。難しい熟語は避け、平仮名多めで。"
    "回答は3〜6文程度の短さを心がけ、結論→理由→具体例 の順で話してください。"
    "参考資料に書かれていないことは推測せず、"
    "分からない場合は『そりゃ、資料に書いてねぇからオラにはわかんねえな』と正直に答えてください。"
)
★ここを変える
雛形の中の ★ で始まる 1 文を 別の指示 に書き換えてみてください。 例: 「語尾を『である調」に」「100字以内で要約して」「受験生の質問に答える気持ちで」

7. まとめ

今日 触れた技術のポイントは 3 つです:

  • RAG の検索は質問の言葉に敏感 — カジュアルな言い回しは、資料の言葉とズレて外すことがある
  • 質問を書き直させると改善する — Query Transformation で AI に検索用のキーワードへ整えさせる
  • 指示の書き方で回答が変わる — システムプロンプト 1 行で AI のキャラクターが動く

どれも、大きな AI(LLM)を、自分たちの用途に合わせて賢く使いこなす ために、 実際の開発現場で使われている工夫です。世の中で華やかに発表される AI サービスの裏側では、 こうした一つひとつの手当てが積み重ねられて、回答の質を作っています。 AI を組み込んだシステムは、こうやって作られていきます。

授業のあとに続きを触れる場所として、『AI 家庭教師 フォローアップ版』 を用意しています。
